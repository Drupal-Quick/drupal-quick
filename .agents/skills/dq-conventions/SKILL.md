# drupal-quick conventions

drupal-quick is a one-shot scaffolding package: it provisions a Drupal site,
generates a custom theme, applies recipes, builds assets, then deletes itself.
Optimize for the **lightest possible footprint** and stay **Drupal-native**
wherever practical.

## The scaffold flow

Three steps (run each through `ddev` under DDEV):

1. `composer exec dq-init` — writes `config.dq.yml` (add `--interactive`, `--ddev`).
2. `composer exec dq-install` — resolves recipes from `templates/recipe-registry.json`.
3. `drush dq:scaffold` — installs Drupal, runs `generate-theme`, applies recipes, builds assets.

Teardown: `drush dq:cleanup` (archives `config.dq.yml`) or `--purge` (deletes it).
Static export: `drush dq:static` (+ `--deploy`). Commands live in
`src/Drush/Commands/DrupalQuickCommands.php`.

## The STARTERKIT token (critical)

The starterkit is `dq_starterkit`. Two rename passes happen during scaffold:

- `drupal generate-theme` rewrites `dq_starterkit` → the new theme machine name
  throughout the **starterkit** files.
- `dq:scaffold` copies recipe `theme-assets/` and does a literal
  `str_replace('STARTERKIT', $themeName, …)` on file **contents and names** for
  text types (`php inc theme twig yml js css md …`).

So in **recipe** theme-assets, write the literal token `STARTERKIT` for anything
that must become the theme machine name — e.g. `STARTERKIT_add_preprocessor(...)`
and helper functions like `_STARTERKIT_article_jsonld()`. In the **starterkit**
itself, write `dq_starterkit_*` (generate-theme renames it).

## Preprocess registration (the dispatch gotcha)

Recipes register preprocess callbacks via the starterkit helper:

```php
STARTERKIT_add_preprocessor('node', function (array &$variables): void { ... });
```

These feed the theme's generic `hook_preprocess()`, which Drupal **always
dispatches under the base hook** (`node`, `page`, `block`) — **never** a
template suggestion such as `node__article`. Registering under a suggestion key
silently never fires.

✅ Register under the base hook and narrow inside the closure:

```php
STARTERKIT_add_preprocessor('node', function (array &$variables): void {
  if ($variables['node']->bundle() !== 'article') {
    return;
  }
  // ...
});
```

❌ `STARTERKIT_add_preprocessor('node__article', ...)` — never runs.

The starterkit itself *can* define real hooks (e.g.
`dq_starterkit_preprocess_node()`) and scope by bundle directly.

## Theme build (Vite + Tailwind v4)

- Source in `starterkits/dq_starterkit/src/`; build emits `dist/main.css|js`.
- CSS is **inlined** into `html.html.twig` (`preprocess_html`), and the linked
  copy is removed in `hook_css_alter()` — don't re-add a `<link>`.
- JS assets in `*.libraries.yml` use `attributes: { type: module }`, NOT
  `type: module` (Drupal reads `type` as the asset kind and rejects `module`).
- Vite dev/HMR: `vite.config.js` writes a `.vite-dev` marker;
  `hook_page_attachments_alter()` swaps `dist/` assets for the live dev server.
  Never commit a `.vite-dev` file.
- Skin/design tokens are merged into the entry `@theme` so config + skins win.

## Light-footprint rules

- Prefer Drupal-native APIs over bespoke logic (e.g. resolve paths via the
  theme/file services, not `getcwd()`).
- Don't add a contrib module when a few lines of hand-built markup will do
  (see the module-free JSON-LD in `docs/structured-data.md`).
- Leave no "generated by drupal-quick" markers in output.
- Keep recipes self-contained: each ships its own `config/` and `theme-assets/`.

## Where things live

- Starterkit theme: `starterkits/dq_starterkit/` (`.theme`, `templates/`, `src/`).
- Recipes: `recipes/<name>/` (`recipe.yml`, `config/`, `theme-assets/`).
- Commands: `src/Drush/Commands/DrupalQuickCommands.php`.
- Config/registry: `templates/config.dq.yml`, `templates/recipe-registry.json`.
- Design notes: `docs/extensibility.md`, `docs/static-deploy.md`, `docs/structured-data.md`.
